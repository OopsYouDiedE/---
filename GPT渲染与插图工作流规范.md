## 0. 本规范在主工作流中的位置

主线：**研究 → 写入 `knowledge/` → 写正文（含 `<!-- [配图] -->` 标记）→ 渲染阶段**。

插图工作流挂在"渲染阶段"内部，但它的**输入依赖**要回溯到 knowledge/ 和正文，不能就地凭空生成：

```
正文里的 [配图] 标记
   │  ①  提参：从正文段落 + knowledge/ 抽取精确技术参数（本规范第 1 节）
   ▼
配图生成规格（Figure Spec）
   │  ②  生成：GPT 出原始图（raw）
   ▼
raw 原始产出 + 联系表（contact sheet）
   │  ③  审核：按类别过检查清单（本规范第 2 节）
   ▼
curated 成品图（命名入库，本规范第 3 节）
   │  ④  接脚本：写进 render_chapterX_full_tex.py 的 FIGURES / 硬编码图位（本规范第 5 节）
   ▼
XeLaTeX 编译 → PDF
   │  ⑤  逐页质检：pdftoppm → PNG，过页面级 SOP（本规范第 4 节）
   ▼
定稿 PDF
```

谁在什么时间点做什么：

- **提参（①）与生成（②）** 在正文定稿后、进入渲染前做。提参必须先于 GPT prompt 存在——**禁止让 GPT 凭"常识"补技术细节**（教训 1）。
- **图级审核（③）** 在图进脚本之前做，逐图过第 2 节清单，不合格打回重生成而不是"先用着"。
- **页面级质检（⑤）** 在每次编译后做，任何一页不过第 4 节 SOP 都必须修后重编。
- 按 AGENTS 第 7 节：多页配图检查、多图审核是无依赖子任务，应拆并行 SubAgent；但"提参 → 生成 → 审核"对单张图是有依赖的串行链，不得强行并行。

## 1. 配图生成流程标准化：从 `[配图]` 标记到 GPT prompt

### 1.1 标记本身的写法（沿用 AGENTS 第 5 节，此处加硬性要求）

`<!-- [配图] 具体描述 -->` 是**内容意图**，不是 prompt。它必须具体到"设计师能执行"，
但进入生成前还要补齐**技术参数层**。描述里以下要素**缺一不可**，缺了就先回正文/knowledge 补，
不许直接丢给 GPT：

| 图类 | 描述必须写全的要素 |
|---|---|
| 动作示范类 | 视角（正/侧/俯）、关键关节角度或对位关系（如"膝盖与脚尖同向""肩髋膝成一线"）、重心位置、发力方向、对错对比时错误点是什么 |
| 数据图表类 | **每一个坐标轴/每一档的具体数值**、单位、数值来源的引用编号、必须落在图内的阈值/红线标签 |
| 场景插画类 | 场景（本土化，见 2.5）、人物关系、动作意图、情绪/调性基调、必须出现的多样性人群 |
| 暴力冲突类 | 尺度上限（不见血、不见打击命中）、"最后手段/创造撤离窗口"的调性锚点、防卫者的撤离方向 |

### 1.2 提参步骤（生成前必做，可复用）

对**每一张**图，生成 prompt 前完成下面四步，产出一份"配图生成规格"：

1. **回查正文对应段落**：找到 `[配图]` 标记所在小节，抄出该小节声明的所有具体数字、
   步骤、适用条件、安全边界。
2. **回查 `knowledge/` 对应文件**：正文覆盖的领域在 knowledge 里必有对应文件
   （AGENTS 第 1 节）。把该文件里与本图相关的技术参数（角度、距离、力值、密度阈值、
   解剖机制）抄进规格。数据类图的**每个数字都要能在 knowledge 里找到出处**，并记下引用编号。
3. **列"图内必现标签"清单**：凡是"看图必须靠文字才看得懂"的信息（两个距离数字、
   五个目标区名称、四个状态名称、密度阈值名称），逐条列出，作为审核项 2.2 的输入。
4. **列"多样性配额"**：本图/本章正文强调了哪些人群（女性、青少年、超重、瘦弱、老人），
   在这一批图里必须至少各出现一次示范者（教训 2）。

> 反例（教训 1 迁移）：描述只写"画一个深蹲动作"就交给 GPT，会得到一个"看起来对"但
> 膝盖内扣/深度不够的姿势。正确做法是先从 knowledge《体能训练与身体掌控》/《女性锻炼者指南》
> 抄出"膝盖与脚尖同向、大腿平行地面、背部中立"再写进规格。

### 1.3 raw → curated

GPT 原始产出统一放 `raw-new/`（或 `raw/`），配一张 `*-contact-sheet.png` 联系表做整批预览
（项目现状已有此惯例，见 `assets/chapter-01/full-render-v1/raw-new/`）。审核通过的图**裁切、
必要时叠加文字标签层**后，另存为正式命名（第 3 节）进产线目录。raw 保留，不进 `\graphicspath`。

## 2. 插图审核检查清单

每张图先过 **2.0 通用项**，再按类别过对应清单。任一"必查"项不过 → 打回重生成/重绘，不许"先用"。

### 2.0 通用（所有图）

- [ ] 图内每个数字/标签都能在正文或 `knowledge/` 找到出处，无 GPT 自造事实（教训 1）。
- [ ] 1.2 第 3 步列出的"图内必现标签"**逐条在图里出现**，不靠图注代偿（图注在正文下方，
      读者盯的是图本身；关键信息不能只活在图注里）。
- [ ] 1.2 第 4 步的"多样性配额"在本批图里已覆盖（教训 2）。
- [ ] 无外国场景/军警/国旗/制服等元素；不确定就本土化（教训 4，详见 2.5）。
- [ ] 无穿帮：多余手指、错误关节数、器械穿模、文字乱码（GPT 中文标签易出乱码，
      宁可后期用矢量文字层叠加，也不要用 GPT 直接烧进图里的中文）。

### 2.1 动作示范类（深蹲、搬运、引体、拖拽、热身等）

- [ ] **解剖学正确**：逐个关节比对 knowledge 里的对位要求。重点复查——
      膝：不内扣、与脚尖同向；髋：屈髋主导而非弓腰；脊柱：中立不圆背；肩髋膝一线类动作的直线关系。
- [ ] **对错对比图**：错误侧要能一眼看出错在哪（红色标注错误关节），正确侧对应绿色/蓝色。
- [ ] **连环分解图**：格与格之间的动作衔接连贯，箭头指示发力/移动方向，手位/脚位清楚。
- [ ] **示范者多样性**（教训 2 核心）：不要一整套动作示范全是青壮年男性。凡正文声称
      "该方法也适合女性/瘦弱者/青少年"的动作，配图示范者就应包含对应人群——否则图会
      反过来削弱正文论点（例：消防员搬运法若只画壮汉，读者不会相信瘦弱者也能用）。
- [ ] 器械动作含安全要素（如深蹲架保护杠位置、失败脱杠方向）与正文安全边界一致。

### 2.2 数据图表类（三级标准、关节冲击、DOMS、距离、密度、五维评估、目标区）

**本类是项目当前最大风险区（见附录 A），审核最严。**

- [ ] **每一个数值都印在图内**，不是只写在图注。参照标杆：`ch01-09-joint-impact-comparison-redesign.png`
      —— 标题、坐标轴刻度、每档具体数值（0.45x / 0.3–0.6x / 2–3x / 7–12x）、结论框、
      数据来源引用编号全部在图内。
- [ ] **安全类阈值"两个数字都要出现"**（教训 3）：凡是描述要求两个及以上关键数字的图，
      少一个即不合格。典型——Tueller 距离图必须**同时**出现 6.4 米（行动红线）与 9.1 米
      （有效反应预警线）；人群密度图必须出现 人/㎡ 数值**且**"行动自主性丧失""压力波出现"
      两个阈值节点。只画一条线/一个色块 = 制造虚假安全感，判不合格。
- [ ] **分类项逐项有文字名称**：五维评估的"武器/人数/体格/位置/训练"、目标区的
      "腹股沟/胫骨/膝侧/鼻梁/耳"、警觉状态的"白/黄/橙/红（或无意识/放松警觉/专注/行动）"
      —— 靠画面无法区分的维度（尤其"训练"这种抽象项）**必须有文字标签**。
- [ ] 机制说明类图（如目标区的神经/骨骼机制）若描述要求配文字说明，则说明文字要在图内或
      紧邻图的结构化标注里，不能只留一个红点让读者猜。
- [ ] 引用编号与正文一致（例：关节冲击图标 [17–19] 要对得上正文该处引用）。
- [ ] 色彩语义与 AGENTS 版式约定一致：红=风险/警告，蓝=重点原则，绿=行动。

### 2.3 场景插画类（章首 hero 图、班组挑战、氛围叙事图）

- [ ] 传达正文的核心情境与情绪，但**不承担数据职责**（数据交给 2.2 的信息图，不要指望
      叙事插画顺带把数字说清——这正是 ch01-03 初版失败、被 redesign 取代的原因）。
- [ ] hero 图人物体现多样性（现状 `ch01-01-hero-flood` 同时有男女老，合格）。
- [ ] 场景本土化，无政治敏感元素（2.5）。

### 2.4 暴力冲突类（威胁应对、徒手拒止、群体协调、持械距离）

面向 10–18 岁，尺度与调性是硬约束：

- [ ] **尺度**：不出现打击命中瞬间、血、伤口特写、面部痛苦特写。目标区图用**临床解剖标注**
      呈现（现状 `ch02-04` 用红点+放大圈，尺度合格），不画"正在打人"。
- [ ] **"最后手段"调性锚点**（教训 4 的调性迁移）：威胁类图必须传达"这是最后手段、
      首选是拉开距离/撤离"，不能拍成鼓励对抗。两个抓手：
      - 画面里防卫者应处于**后撤/拉开距离**姿态，配撤离方向箭头（现状 ch02-03/ch02-02
        防卫者都在后撤，合格）。
      - **图注统一补最后手段措辞**：如"只用于创造撤离窗口"。现状只有 `ch02-04` 图注有此措辞，
        其余威胁图图注偏中性 → **要求所有威胁类图注都加此类框定**。
- [ ] 群体协调类图要突出"保护弱者/协同疏散"而非"围殴"：优先画疏散老人小孩、用背包/椅子
      当盾牌等防御性动作（现状 ch02-05 合格且顺带体现了多样性）。
- [ ] 武器只作威胁标识（如对方持刀/持棍），不美化、不做特写。

### 2.5 政治边界（教训 4，从严执行，适用所有类别）

- [ ] 无外国国旗、军警制服、可辨识的外国政府/军事符号。
- [ ] 涉及军警/救援/对抗的场景，人物着装本土化、中性化（现状 ch01/ch02 用改良汉服/唐装，
      规避了制服问题，是正确做法）。
- [ ] 即使正文属于"自然灾害应急"等例外范畴、文字未明令禁止，**插图仍按从严原则**——
      宁可用更中性/本土的视觉，也不引入可能触线的外国元素（对应釜石奇迹案例即便属例外
      仍被判违规的先例）。

## 3. 版本迭代与命名规范

### 3.1 现状观察（据 `assets/` 实际文件推断）

项目现在同时存在多套命名，容易混淆，本节先固化解读、再给出向前约定：

| 现存形态 | 含义（推断） | 是否进产线 |
|---|---|---|
| `full-render-v1/`、`full-render-v3/` | 整章渲染批次目录，`vN` 是**整批**版本 | 是（`\graphicspath` 指向它） |
| `chXX-NN-slug.png` | 产线成品图：`NN`=图位序号，`slug`=英文短名 | 是 |
| `chXX-NN-slug-redesign.png` | 同一图位因**信息缺失被重绘**的数据化版本（见下） | 是（脚本引用 redesign） |
| `chXX-NN-slug.png`（无 redesign 后缀，但存在 redesign 同名） | 被 redesign 取代的**旧版**，未删 | 否（死资产） |
| `modern-hanfu-v3/` | **风格探索**批次（试服装/画风），含 `*-style-page-proof` | 否 |
| 顶层 `chXX-slug-v1.png`（5 个） | 早期单图试作 | 否 |
| `raw-new/raw-NN.png` + `*-contact-sheet.png` | GPT **原始产出**与联系表 | 否 |

**redesign 的真实成因（重要）**：对比 `ch01-03-three-level-standard.png`（初版）与
`ch01-03-...-redesign.png`（终版）——初版是氛围叙事画（一个人跑步→拖人→废墟前单膝跪），
**丢光了全部量化标准**；redesign 是结构化信息图，把 5 公里/10 层楼梯/70 公斤拖拽 10 米/
20 分钟等标准和"14 岁以下年龄修正"脚注全部标了出来。`ch01-09` 同理。
**结论：redesign 不是审美迭代，是"初版把 knowledge 里的数字画丢了、被迫数据化重绘"。**
这正是教训 1/3 的视觉版——所以第 2.2 节把"数字必须印进图内"列为最严审核项。

### 3.2 向前命名约定

- 产线图：`chXX-NN-slug.png`，放当前批次目录 `chapter-XX/full-render-vN/`。
- 整批重渲染（换画风、换整章版式）→ 新目录 `full-render-v(N+1)/`，旧目录整体保留一版即可，
  不需长期堆叠三四个版本。
- 单图重绘 → **不要**用 `-redesign`/`-v2` 这种语义模糊的后缀堆叠；改用
  `chXX-NN-slug-r2.png`（r=revision，`r2`/`r3` 递增），并在**图位清单**（见 3.3）里记一行
  "为什么重绘"。老版本移入同目录的 `deprecated/` 子目录，不与产线图混放。
- 风格试作、raw、联系表：一律留在 `raw/` 或带明确后缀，永不进 `\graphicspath`。

### 3.3 何时必须重生成一张图（触发条件）

命中任一条即必须重做，不许将就：

- 数据类图缺任一必现数字/阈值/分类标签（2.2）。
- 动作类图姿势解剖学错误（2.1）。
- 正文对应段落的数字/步骤/适用条件发生变化，图与正文**不再一致**。
- 示范者人群与正文多样性主张脱节（2.2 / 教训 2）。
- 出现政治/尺度/调性违规（2.4 / 2.5）。
- 纯审美不满意但信息正确：**低优先**，可排后，不阻塞出版。

### 3.4 图位清单（新增，强制）

每章维护一份"图位清单"（可放章节进展 memory 或 `assets/chapter-XX/FIGURES.md`），
一图一行：图位序号 / slug / 当前采用文件名 / 对应正文行号或小节 / 数据来源引用编号 /
最近一次重绘原因。**作用**：防止死资产失控（现状 ch01 已有约 30 个未进产线的旧版/风格/raw
文件散落），并让"哪张图为什么是现在这版"可追溯，避免被后续"精简"抹掉（教训 5）。

## 4. PDF 页面级质检 SOP

流程：编译 PDF → `pdftoppm -r 150 -png xxx.pdf page` 转逐页 PNG → 逐页过下表。
任一项不过 → 修脚本/挪模块后重编。

### 4.1 逐页必查（整合 AGENTS 第 6 节 + 实测补充）

| # | 检查项 | 判据 | 修法 |
|---|---|---|---|
| 1 | 标题孤立 | 小节标题落在页尾、正文被推到下一页 | 脚本已用 `\Needspace{N\baselineskip}`，命中则调大 N |
| 2 | 段落硬拆 | 一段在页内被拆成两半、语义断裂 | 宁可整段挪下一页；`\Needspace` 或允许上方图浮动 |
| 3 | 提示框断裂 | callout（注意/要点/行动）跨页断开 | callout 前的 `\Needspace{7\baselineskip}` 调大 |
| 4 | 表格溢出 | 表格超出版心宽/被截断 | 用 `tabularx` 的 `X` 列自适应；必要时 `\footnotesize` |
| 5 | 图片含义不清 | 图缺必现标签、缩放后文字看不清、对错分不清 | 回第 2.2 节重做图，不是靠排版补救 |
| 6 | **页尾大面积空白** | 页面下部留白 ≳ 30% 且非章末 | 见 4.2，**实测 ch02 p6 命中此项** |
| 7 | 图注/引用排版 | 图注居中、引用小字紧凑、无编号错乱 | 脚本 `captionsetup{labelformat=empty}` 已处理 |
| 8 | 页眉页脚 | 章名、页码正确 | 脚本 `fancyhdr` 已处理 |

### 4.2 关于"页尾大面积空白"（实测重点）

成因：`\begin{figure}[H]` 强制图就地、加上 `\Needspace`，会把"放不下的图或紧跟的标题"
整块推到下一页，于是上一页尾部大片留白。**这是本项目最容易复发的页面级缺陷**。
处理优先级：

1. 首选**缩小该图的 `height`**（脚本里 `height=0.34\textheight` 等）让它能挤进本页尾部；
2. 或把图上方的 callout/短段落**语义无损地重排**填补空白；
3. 或对该图放宽为可浮动（去掉 `[H]` 改 `[htbp]`）——仅在不破坏"图紧跟其解释段落"时用；
4. 只有"整块模块下移"是最优解时才接受留白（AGENTS 允许"宁可整块下移"），
   但"下移后留 40% 空白"不算最优解，要先试 1–3。

### 4.3 插图内容在 PDF 里的复检

页面质检时**顺带复看插图本身**（有些问题只有在正文语境里才暴露）：

- [ ] 图与它上下文的正文数字一致（图注 6.4/9.1 米，图里也要有）。

## 5. 与渲染脚本对接的机械性检查（易被忽视的坑）

渲染脚本（`render_chapter1_full_tex.py` / `render_chapter2_full_tex.py`）用**关键词子串匹配**
把 `[配图]` 标记和 callout 映射到图/样式。这套机制有两个脆弱点，必须纳入检查：

### 5.1 关键词漂移会让图/提示框"静默消失"

- `FIGURES = [(关键词, 文件名, 图注), ...]`，`render_comment()` 用
  `if 关键词 in 配图行` 命中才出图。**若正文改动措辞导致关键词不再是配图行的子串，
  这张图会无报错地消失**，PDF 少一张图但编译不报错。
- `CALLOUTS` 同理：提示框分类靠关键词命中，正文改词 → 提示框退化成普通段落，静默丢失。
- **规则**：任何改动正文 `[配图]` 行或提示框首句后，必须**回脚本核对每个 key 仍能子串命中**；
  或编译后数一遍 PDF 里的图数量/提示框数量与预期是否相等。建议在脚本 `main()` 里加一句
  断言：命中的图位数 == `FIGURES` 里非空文件名条目数，不等就报错。

### 5.2 空文件名 = 故意不出图

- `FIGURES` 里文件名留空（如 ch01 的"4 周训练计划周视图"），表示脚本**故意用表格代替图**
  （`render_week_plan`）。这是 AGENTS 第 6 节"图片是否改成图表由脚本决定"的合法用法，
  不是漏配。图位清单（3.4）里要注明"此图位已转表格"，避免有人误以为图丢了又补一张。

### 5.3 硬编码图位

- hero 图、班组挑战图不在 `FIGURES` 里，而是在标题处理分支**硬编码**插入
  （ch01：`## 一、` 后插 `ch01-01-hero-flood`；`#### 班组挑战` 后插 `ch01-20`）。
  改这些锚点标题时要同步改脚本，否则 hero/挑战图会消失。

### 5.4 `\graphicspath` 与死资产

- 脚本 `\graphicspath` 只指向 `chapter-XX/full-render-vN/`，**只有该目录下的文件会被找到**。
  其余目录（`modern-hanfu-v3/`、顶层 `chXX-*-v1.png`、`raw-new/`、被 redesign 取代的旧版）
  都是死资产，不影响渲染但会积累混乱。按 3.2/3.4 用 `deprecated/`、`raw/` 归拢并登记。
